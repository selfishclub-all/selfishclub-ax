## 규칙
사용자가 "풀해줘" / "메인 변경사항 풀해줘" 라고 할 때:
1. **기본: 현재 브랜치(`feat/sharing-crm-team` 등)를 풀** — 다른 맥북에서 작업한 내용이 거기 있음
2. main을 명시하지 않는 한 main은 거의 안 건드림 (main은 merge 시에만 바뀜)
3. 세션 로그/메모리(`.omc/*`, `memory/*`, `.claude/settings.local.json`)가 로컬에서 미커밋 상태로 바뀌어 있을 때:
   - 로컬 버전은 이번 세션에서 자동 생긴 드리프트 (하네스가 자동 업데이트)
   - 원격 버전은 다른 맥북에서 사용자가 수동으로 커밋한 최신본
   - **원격(풀 받은) 버전을 우선 채택, 로컬 드리프트는 버림** (stash drop)

## Why
집맥북 ↔ 회사맥북을 오가며 작업함. GitHub는 두 맥북의 중간 통로.
세션 로그/메모리가 `.gitignore`에 빠져있어서 추적됨(의도적) — 양쪽 맥북이 동일 컨텍스트 유지하기 위함.
로컬에서 자동 업데이트된 상태 파일은 이번 세션의 일시적 로그라서, 다른 맥북의 커밋된 버전이 항상 "정답".

## How to apply
- "풀해줘" → `git fetch` → 현재 브랜치 원격과 비교 → `git stash push -- <상태파일들>` → `git pull --ff-only` → `git stash drop` (pop 아님, drop으로 버리기)
- 충돌 발생 시: 원격 버전 유지(`git checkout HEAD -- <파일>`), 로컬 변경은 폐기
- 사용자에게 받은 커밋 개수/메시지 간단히 보고
- 복잡하게 AskUserQuestion 띄우지 말 것 — 사용자가 매번 같은 결정 내리는 게 번거로움

## 예외
- 로컬에서 **의도적으로** 편집한 파일(코드/PRD/에이전트 md)은 stash 대상 아님 — 이건 충돌 시 반드시 사용자에게 확인
- 감지 방법: git diff에서 `.omc/`, `memory/` 외의 파일이 수정됐는지 체크
