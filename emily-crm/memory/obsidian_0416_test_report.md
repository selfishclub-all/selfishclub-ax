# 알림톡 크론 테스트 리포트 (4/14~4/16)

## Test 1 — 발송직전 DB 조회 (4/14) ✅

### 뭘 검증했나
승인 시점에는 미신청자(0명)인데, 발송 시점에 신청자로 전환되면 카톡이 가는지.

### 흐름
```
크론 → 슬랙 미리보기 → 에밀리 승인 → SQL 재생성(승인 시점 SQL) → Wait(발송시간 대기)
→ 취소 확인 → **Postgres 재조회(발송 직전)** → 발송 데이터 → SOLAPI 발송
```

### 핵심 변경: 발송 직전 DB 조회
```
[기존] SQL재생성 → Postgres조회 → 발송데이터 → Wait → Switch
[변경] SQL재생성 → Wait → 취소확인 → Postgres조회 → 발송데이터 → Switch
```
- 승인 시점이 아니라 **발송 직전에 Postgres 조회** → 그 사이에 신청한 사람도 포함
- `Code: SQL 재생성`에 scheduledDate 계산 + 파싱 데이터 패스스루
- `Code: 취소 확인`에서 staticData._currentSend에 템플릿 데이터 저장
- `Code: 발송 데이터`가 _currentSend에서 템플릿 정보 읽음

### 테스트 결과
- 템플릿: ③ D-1, iid_999, eventdate=4/15
- 크론: `0 35 4 * * *` (KST 17:35)
- 에밀리 승인(17:37) → 미신청자 → 신청자 전환 → Wait 해제 → Postgres 재조회 → 1명 → 카톡 ✅

### 삽질
- 크론 시간을 UTC로 계산해서 3번 실패 (n8n 서버는 **EDT**)
- 백업(0413)에서 복원했더니 이전 코드 복원됨 → 취소 확인 버전 체크 비활성화해서 우회
- execution에서 Postgres 1명 나왔는데 발송 0건 → 버전 체크 실패 (`_ver` undefined) → 주석처리

## Test 2 — 취소 → 재승인 중복방지 (4/16) ✅

### 뭘 검증했나
승인 → 취소 → 재승인 했을 때 카톡이 **1번만** 오는지.

### 배경: 중복 발송 버그 (4/13 발견)
```
승인(A) → Wait 시작(A) → 취소 → cancel 플래그 ON
→ 재승인(B) → cancel 플래그 OFF (재승인이 삭제) → Wait 시작(B)
→ Wait(A) 해제 → cancel 플래그 없음 → 발송됨 ❌ (중복!)
→ Wait(B) 해제 → 발송됨 (정상)
= 카톡 2번 발송
```

### 수정: cancel 플래그 → 버전 카운터
```javascript
// Code: SQL 재생성 — 승인 시 ver +1
const verKey = `ver_${d.t}_${d.iid}`;
staticData[verKey] = (staticData[verKey] || 0) + 1;
const _ver = staticData[verKey];
return [{ json: { recipients_sql: sql, scheduledDate, _ver, ...d } }];

// Code: 취소 준비 — 취소 시 ver +1
const verKey = `ver_${d.t}_${d.iid}`;
staticData[verKey] = (staticData[verKey] || 0) + 1;

// Code: 취소 확인 — Wait 끝난 후 ver 비교
const verKey = `ver_${d.t}_${d.iid}`;
const currentVer = staticData[verKey] || 0;
if (d._ver !== currentVer) return [];  // 버전 불일치 → 발송 차단
```

### 동작 원리
```
승인(A): ver=1, _ver=1 → Wait(A)
취소:    ver=2
재승인(B): ver=3, _ver=3 → Wait(B)
Wait(A) 해제: _ver=1 ≠ ver=3 → 차단 ✅
Wait(B) 해제: _ver=3 = ver=3 → 발송 ✅
= 카톡 1번만
```

### 테스트 결과
- 흐름: 수정 → 승인 → 취소 → 수정 → 재승인 → **카톡 1건만 도착** ✅
- 환경: iid_999, eventdate=4/17, 템플릿 ③ D-1
- 크론: KST 10:08 (`0 8 21 * * *` EDT)
- 실행 로그: 11259(trigger) → 11260~11265(webhook, 전부 success)

### 삽질: 취소 확인 노드 이중 이스케이프

**첫 시도 (KST 09:47)**: 크론 정상 실행, 슬랙 미리보기 도착, 승인 클릭 → **카톡 안 옴**

**원인**: `Code: 취소 확인` 노드의 템플릿 리터럴이 이중 이스케이프됨
```javascript
// 저장된 코드 (잘못됨)
const verKey = \`ver_\${d.t}_\${d.iid}\`;
// → JavaScript 문법 에러 → 노드 실행 실패

// 정상 코드
const verKey = `ver_${d.t}_${d.iid}`;
```

**왜 이중 이스케이프?**: API PUT으로 jsCode 업데이트할 때 bash heredoc/jq가 백틱과 `${}`를 추가 이스케이프함

**해결**: python3 `json.dump()`로 payload 생성 → PUT

**교훈**: n8n Code 노드 jsCode를 API로 업데이트할 때는 **반드시 python으로 JSON 생성**. bash/jq 금지.

## 크론 시간 규칙 (확정)

### n8n 서버 타임존
- **America/New_York (EDT, UTC-4)** — UTC 아님!
- KST → EDT 변환: **KST - 13시간** (서머타임 KST - 14시간)

### 크론 포맷: 6자리 (초 포함)
```
초 분 시(EDT) 일 월 요일
```

### 예시
| KST | EDT | 크론 |
|-----|-----|------|
| 10:00 | 21:00 | `0 0 21 * * *` |
| 13:00 | 00:00(+1) | `0 0 0 * * *` |
| 15:00 | 02:00(+1) | `0 0 2 * * *` |
| 17:45 | 04:45(+1) | `0 45 4 * * *` |
| 22:00 | 09:00(+1) | `0 0 9 * * *` |

### 주의사항
- `0 21 * * *` (5자리) 쓰면 "매시 21초"로 해석됨 → 반드시 6자리
- API로 크론 수정 후 **deactivate → activate** 필수 (스케줄러 재등록)
- 테스트 시 매분 크론(`0 * * * * *`)으로 먼저 검증 → 복원

### 프로덕션 크론 스케줄 (확정)
| # | 알림톡 | KST | EDT 크론 |
|---|--------|-----|---------|
| ① | 오픈알림 | 15:00 | `0 0 2 * * *` |
| ② | 오픈리마인드 | 13:00 | `0 0 0 * * *` |
| ③ | D-1 리마인드 | 13:00 | `0 0 0 * * *` |
| ④ | 당일리마인드 | 14:00 | `0 0 1 * * *` |
| ⑤ | 할인쿠폰 | 13:00 | `0 0 0 * * *` |
| ⑥ | 입장링크 | 19:00 | `0 0 6 * * *` |
| ⑦ | 시작알림 | 20:00 | `0 0 7 * * *` |
| ⑧ | 혜택안내 | 22:30 | `0 30 9 * * *` |

> 실제로는 **단일 크론** `0 0 21 * * *` (KST 10:00)이 모든 템플릿을 판별하고, scheduledTime으로 예약 발송. 위 시간은 SOLAPI sendAt에 들어가는 발송 시간.

### 삽질에서 배운 것

#### 1. /tmp 파일 재사용 금지
이전에 저장한 /tmp 파일로 PUT하면 그 시점의 설정으로 되돌아감.
**항상 GET → 수정 → PUT.**

#### 2. PUT에 staticData 포함하면 버전 카운터 덮어씌워짐
승인 시 ver 카운터 증가 → PUT에 이전 staticData 포함 → 카운터 리셋 → `_ver !== currentVer` → 발송 차단.
**코드/크론만 변경할 때는 staticData 제외: `{name, nodes, connections, settings}`만.**

#### 3. Wait 중 deactivate 금지
워크플로우 deactivate하면 Wait 대기 중인 실행이 전부 종료.
**발송 대기 중에는 절대 deactivate 하지 않기.**

#### 4. 자동승인 로직 함부로 비활성화 금지
`approve_direct` 리턴을 주석처리하면 ⑥⑦ 승인 경로가 깨짐.
**자동승인 테스트 시 staticData 값(zoomUrl/benefits)을 삭제하는 방식 사용.**
